# Context Governance — מדריך למשתמש

> **מי צריך לקרוא את זה:** אתה — בעל הפרויקט / המפתח שעובד עם Claude Code.
> **מסמך מקביל לסוכנים:** `~/.claude/docs/GOVERNANCE-AGENT-GUIDE.md`
> **גרסה:** 1.0 (2026-04-12)

---

## מה זה ולמה זה קיים

Context Governance הוא מערכת ניהול מידע והקשר שפועלת **אוטומטית** בכל סשן של Claude Code. היא נועדה לפתור 5 בעיות שגורמות לרגרסיות, בלבול, ואיבוד עבודה:

**בעיה 1 — "מאיפה התחלנו?"**
בלי governance: כל סשן חדש שואל "מה עשינו בפעם האחרונה?" ומבזבז 10 דקות על re-discovery.
עם governance: ה-bootstrapper מציג briefing אוטומטי תוך 5 שניות.

**בעיה 2 — "מי דרס את מי?"**
בלי governance: שני סשנים מקבילים כותבים ל-PLAN.md ודורסים אחד את השני.
עם governance: file-locking + parallel-session-merge מגלים ומיישבים קונפליקטים.

**בעיה 3 — "סיימנו אבל שברנו משהו אחר"**
בלי governance: Claude אומר "done" בלי להריץ בדיקות.
עם governance: Verification Gate חוסם "done" בלי evidence חיצוני.

**בעיה 4 — "למה החלטנו ככה?"**
בלי governance: החלטות מתועדות רק בהיסטוריית שיחה שנמחקת ב-compact.
עם governance: MEMORY.md שומר החלטות עמידות, lessons learned, ו-pitfalls.

**בעיה 5 — "איפה הקובץ הנכון?"**
בלי governance: מידע מפוזר ב-10 קבצים שחלקם סותרים.
עם governance: CONTEXT-MANIFEST.md הוא אינדקס-על שאומר מה authoritative, מה stale, ומה archived.

## מה נכנס לתוקף אוטומטית

ברגע שהמערכת מותקנת, הדברים הבאים קורים **בלי שתבקש**:

**בתחילת כל שיחה:**
- Claude קורא את קבצי ההקשר הקריטיים
- מציג briefing קצר: "מצב נוכחי / פעולה הבאה / סיכונים"
- אתה רואה ~3-5 שורות בתחילת התגובה הראשונה

**לפני כל עריכת קוד:**
- נבנית מפת השפעה (impact map) — אילו קבצים מושפעים
- אם הסיכון גבוה (קובץ קריטי) — Claude עוצר ושואל אישור
- אם הסיכון נמוך — עובר שקוף

**אחרי כל milestone:**
- PLAN.md מתעדכן ברקע
- MEMORY.md מקבל רשומה אם הייתה החלטה חשובה
- אתה לא צריך לעשות כלום

**בסוף כל שיחה:**
- HANDOFF.md נכתב אוטומטית
- הסשן הבא יקרא אותו ויידע מאיפה להמשיך

## מה עדיין דורש ממך פעולה ידנית

- **`/full-finish`** — release pipeline (git push, GitHub release). תמיד ידני.
- **`/context-governance full`** — ביקורת עומק. תמיד ידנית (פרט ל-cron שבועי).
- **Stop-Report** — כל פעם שיש סתירה, Claude עוצר ושואל. זה פיצ'ר, לא באג.
- **פעולות git הרסניות** — push --force, reset --hard. תמיד דורשות אישור.

## הגנה מפני drift של סשנים מקבילים (2026-04-17)

אם אתה מריץ יותר מסשן Claude אחד במקביל על אותו פרויקט, יכולות להיווצר בעיות — סשן אחד כותב handoff, השני לא יודע ומשכתב אותו. כדי למנוע את זה נוסף הסקיל `/pre-close-check`.

**איך זה עובד:** לפני כל סגירת סשן (`/full-finish`), כתיבת handoff (`/live-state-orchestrator`), או סיום תכנית (`/plan-and-execute` שלב 3.3) — הסקיל מבצע 5 בדיקות אוטומטיות:
1. git log של השעה האחרונה (האם סשן אחר ביצע commits?)
2. mtime של קבצי handoff (מי נגע לאחרונה?)
3. version.json ↔ CLAUDE.md ↔ HANDOFF pointer (האם כולם מסכימים על הגרסה?)
4. ספירת handoff פעילים (חייב להיות בדיוק 1)
5. סינכרון CONTEXT-MANIFEST עם המצב בפועל

אם נמצאה פעילות מקבילית או drift, Claude **עוצר** לפני הכתיבה ושואל אותך איך לפתור (merge? abort? override?).

**למה זה חשוב:** תקרית 2026-04-17 (v1.4.112-b/v1.4.113) — סשן אחד כתב handoff של v1.4.112-b כפעיל, בזמן שסשן מקבילי שחרר v1.4.113. בלי ה-check הזה, drift כזה לא מתגלה עד סשן חדש שמבחין בסתירה.

## מבנה הקבצים

כל פרויקט מנוהל מכיל את המבנה הזה:

```
<שורש הפרויקט>/
│
├── CLAUDE.md                    ← "מי אני ואיך אני עובד" (קצר, <500 שורות)
│
├── docs/context/                ← תיקיית ההקשר המרכזית
│   ├── CONTEXT-MANIFEST.md      ← אינדקס-על: כל קובץ, תפקידו, סטטוס
│   ├── HANDOFF.md               ← גשר בין סשנים (מי עשה מה, מה הבא)
│   ├── OPEN-PROBLEMS.md         ← רשימת באגים/בעיות פתוחות
│   ├── MEMORY.md                ← החלטות עמידות, לקחים, מלכודות ידועות
│   ├── CONVENTIONS.md           ← כללי קוד, אבטחה, סגנון
│   ├── SCHEMAS-INDEX.md         ← אינדקס של סכמות, API, מודלים
│   └── GOTCHAS.md               ← כללים שנולדו מבאגים (מספור רציף, לעולם לא מוחקים)
│
└── Plans/
    └── PLAN.md                  ← תוכנית העבודה הנוכחית (sub-plans, milestones, סטטוס)
```

## היררכיית מקורות אמת

כשיש סתירה בין שני קבצים — מי מנצח?

1. **הקוד עצמו + הבדיקות** — תמיד נכון (זו המציאות)
2. **version.json** — הגרסה הרשמית
3. **PLAN.md** — מה שאנחנו עובדים עליו עכשיו
4. **OPEN-PROBLEMS.md** — מה שבור
5. **HANDOFF.md** — מה שהסשן האחרון עשה
6. **MEMORY.md** — החלטות היסטוריות
7. **CONVENTIONS.md** — כללי קוד
8. **CLAUDE.md** — הנחיות כלליות

**ככל שהמספר קטן יותר — הוא authoritative יותר.**

## מחזור חיים של HANDOFF

HANDOFF הוא ה"פתק" שסשן אחד משאיר לסשן הבא.

```
[active] ←── סשן נוכחי קורא אותו
    │
    ▼
[consumed] ←── התוכן הועבר ל-PLAN.md / MEMORY.md
    │
    ▼
[archived] ←── לא רלוונטי יותר, שמור להיסטוריה
```

**כלל ברזל:** רק HANDOFF אחד יכול להיות `active` בכל רגע נתון. אם נמצאים שניים — זה red flag ש-Claude יעצור ויתריע.

## Verification Gate — מה זה ולמה זה חוסם

לפני שClaude מכריז "סיימתי", הוא חייב להציג **ראיה חיצונית** שהעבודה עובדת:
- פלט בדיקה (test runner) עם PASS
- פלט build/lint עם exit code 0
- docker logs שמראים את ההתנהגות החדשה
- HTTP probe עם קוד תגובה צפוי
- screenshot של UI עובד

"בדקתי בעיניים ונראה לי בסדר" — **לא מתקבל.**

## Kill Switch — איך לכבות

אם משהו משתבש או חוסם אותך:

```bash
export GOVERNANCE_HOOKS=0
```

זה משבית את כל ה-hooks באופן מיידי. Claude חוזר להתנהגות רגילה. לא צריך לערוך אף קובץ. כדי להפעיל מחדש:

```bash
unset GOVERNANCE_HOOKS
```

## שאלות נפוצות

**"למה Claude שואל אישור לפני עריכה?"**
כי הקובץ מסומן HIGH risk. זה מונע רגרסיות שקטות. אפשר לאשר בלחיצה.

**"למה יש briefing בתחילת כל שיחה?"**
זה ה-bootstrapper שטוען הקשר. בלעדיו Claude לא יודע מה המצב הנוכחי ומתחיל מאפס.

**"למה Claude עוצר ואומר Stop-Report?"**
הוא מצא סתירה שלא יכול לפתור לבד. צריך את ההכרעה שלך. זה מונע הזיות.

**"מה קורה אם אני רוצה לעבוד בלי governance?"**
`export GOVERNANCE_HOOKS=0`. או פשוט אל תריץ `/init-governance` — Claude יעבוד רגיל על פרויקטים שלא מאותחלים.

**"האם זה עובד על כל פרויקט?"**
כן. ה-skills גנריים. הם מזהים דינמית את המבנה של כל פרויקט דרך CONTEXT-MANIFEST.md. אין שום hardcoding.
